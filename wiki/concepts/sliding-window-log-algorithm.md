# 슬라이딩 윈도우 로그 (Sliding Window Log Algorithm)

## 한 줄 정의

요청 **타임스탬프 자체를 모두 저장**하고, 매번 현재 윈도우 밖의 오래된 타임스탬프를 제거해 정확한 윈도우 내 요청 수를 유지하는 처리율 제한 알고리즘 (ch04, p.64-65).

## 왜 필요한가

[[fixed-window-counter-algorithm]]의 경계 burst 문제를 **완전히** 해결한다. 어떤 1분 롤링 윈도우를 보더라도 임계치를 초과하지 않음을 보장.

## 핵심 메커니즘

타임스탬프는 보통 [[redis]] **sorted set**에 저장한다.

1. 요청 도착 시각을 가져옴.
2. **윈도우 시작 이전의 모든 타임스탬프 제거**.
3. 새 타임스탬프 추가.
4. 로그 크기 ≤ 허용 카운트면 통과, 아니면 reject (단, 추가는 유지).

### 책 예시 (Figure 4-10, 분당 2 요청 허용)

- 1:00:01 도착 → log=[1:00:01], size 1 ≤ 2 → 통과.
- 1:00:30 도착 → log=[1:00:01, 1:00:30], size 2 ≤ 2 → 통과.
- 1:00:50 도착 → log=[1:00:01, 1:00:30, 1:00:50], size 3 > 2 → **reject**, 그래도 1:00:50은 로그에 남음.
- 1:01:40 도착 → 윈도우 시작은 1:00:40 → 1:00:01·1:00:30 제거 → log=[1:00:50, 1:01:40], size 2 ≤ 2 → 통과.

## 트레이드오프

**Pros**
- **매우 정확**. 어떤 롤링 윈도우에서도 임계치 초과 없음.

**Cons**
- **메모리 비용 큼**: rejected 요청의 타임스탬프도 일정 시간 남아 있음. 트래픽이 많을수록 메모리·CPU(원소 제거) 비용 증가.
