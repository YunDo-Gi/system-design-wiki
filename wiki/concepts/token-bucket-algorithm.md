# 토큰 버킷 알고리즘 (Token Bucket Algorithm)

## 한 줄 정의

용량이 정해진 버킷에 토큰이 일정 속도로 보충되고, 요청은 토큰 1개를 소모하는 방식의 처리율 제한 알고리즘. **버스트를 허용**하는 것이 특징 (ch04, p.59-61).

## 왜 필요한가

가장 널리 쓰이는 알고리즘. 메모리 효율이 좋고, **단기 버스트를 자연스럽게 허용**하면서도 평균 속도는 제한 가능. Amazon·Stripe API throttling이 채택.

## 핵심 메커니즘

두 파라미터:

- **Bucket size**: 버킷에 담을 수 있는 최대 토큰 수.
- **Refill rate**: 초당 채워지는 토큰 수.

동작 (ch04, p.59):

1. Refiller가 일정 속도로 버킷에 토큰 추가. 가득 차면 overflow 폐기.
2. 요청 도착 시 버킷에 토큰이 있으면 1개 소모 후 통과.
3. 토큰 없으면 요청 drop (또는 큐잉).

### 책 예시 (Figure 4-6, 4 tokens · refill 4/min)

- 1:00:00: 토큰 4 → 요청 1 통과, 토큰 3.
- 1:00:05: 토큰 3 → 동시 요청 3개 모두 통과, 토큰 0.
- 1:00:20: 토큰 0 → 요청 drop.
- 1:01:00: 1분 후 refill로 토큰 4 복귀.

### 버킷 단위 (ch04, p.61)

- API 엔드포인트별 별도 버킷이 일반적 (post / friend-add / like가 각각 별도 정책).
- IP 기반 차단이면 IP당 1개.
- 글로벌 한도(예: 시스템 전체 10,000 RPS)에는 전역 단일 버킷.

## 트레이드오프

**Pros**
- 구현 단순, 메모리 효율적.
- **단기 버스트 허용** — 토큰이 남아 있는 동안은 통과시킴. flash sale 같은 정상 급증에 유리.

**Cons**
- 두 파라미터(크기·refill rate)를 비즈니스에 맞게 튜닝하는 게 의외로 어려움.
