# URL Redirection — 301 vs 302

## 한 줄 정의

HTTP redirect 응답으로 클라이언트를 다른 URL로 보내는 방식. **301(Moved Permanently)** 과 **302(Found, 임시)** 중 선택이 URL 단축기 설계의 핵심 트레이드오프다 — 서버 부하 절감 vs 클릭 분석 가능성 (ch08, p.123).

## 왜 필요한가

[[url-shortener]]의 리다이렉트 경로는 단축 코드 요청을 받아 원본 URL로 브라우저를 보낸다. 둘 다 `Location` 헤더에 원본 URL을 실어 보내지만, **브라우저의 캐싱 동작이 정반대**라 시스템 부하와 분석 능력이 갈린다.

## 핵심 메커니즘

```
GET /abc123 HTTP/1.1
        ↓
HTTP/1.1 301 Moved Permanently      HTTP/1.1 302 Found
Location: https://original.url/...   Location: https://original.url/...
```

| | **301 Moved Permanently** | **302 Found (임시)** |
|---|---|---|
| 의미 | 영구 이동 | 임시 이동 |
| 브라우저 캐싱 | **캐싱함** — 이후 같은 단축 URL은 서버를 안 거치고 직접 원본으로 | **캐싱 안 함** — 매 요청 서버 경유 |
| 서버 부하 | **낮음** (첫 요청 후 서버 우회) | 높음 (매번 서버 도달) |
| 클릭 분석 | **불가** (서버가 후속 클릭을 못 봄) | **가능** (모든 클릭이 서버 통과) |
| SEO | 링크 가치 원본으로 이전 | 이전 안 됨 |

## 트레이드오프 & 선택 기준

- **서버 부하 최소화가 최우선** → 301. 트래픽이 막대하고 클릭 통계가 불필요한 순수 리다이렉터에 적합.
- **클릭 분석이 비즈니스 가치** → 302. 마케팅 단축 링크(캠페인 클릭 추적), A/B 테스트, 과금 기반 단축 서비스는 302 필수. 매 클릭을 서버가 봐야 집계 가능.
- 책의 기본 권고는 분석 가치를 고려해 상황에 따라 선택. 많은 상용 단축기(Bitly 등)는 **분석이 핵심 상품**이라 302 계열을 쓴다.

## 실무 적용 시 고려사항

- **301 캐싱의 비가역성**: 한번 301로 응답하면 브라우저·중간 프록시가 오래 캐싱한다. 원본 URL이 바뀌거나 단축 코드를 재활용해야 하면 곤란 → 단축 매핑이 immutable이 아니면 301 위험.
- **분석 + 부하 절충**: 302로 받되 서버단은 캐시([[caching-strategies]])로 원본 조회를 받아내 DB 부하를 낮춘다. "매 요청 서버 경유"의 비용을 캐시로 흡수.
- **307/308**: 메서드 보존이 필요한 API redirect에는 307(임시)·308(영구)이 더 정확. 단축기는 GET이라 301/302로 충분.
- **knot 적용**: knot은 redirect 엔드포인트에 [[rate-limiting]]을 이미 적용. 분석을 원하면 302 + 캐시 조합.

## 다른 개념과의 관계

[[caching-strategies]]와 직접 엮인다 — 301은 브라우저가 캐시를 대행, 302는 서버가 캐시를 책임진다. [[url-shortener]]의 읽기 우세(10:1) 특성이 이 선택의 배경.
