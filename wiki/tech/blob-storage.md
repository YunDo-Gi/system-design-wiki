# Blob / Object Storage

## 한 줄 정의

**대용량 비정형 바이너리(BLOB — Binary Large Object)**를 단일 엔티티로 저장하는 시스템. 영상·이미지·백업처럼 큰 파일을 저렴하게·사실상 무한히 확장 가능하게 보관한다 (ch14, p.225). 대표: AWS S3, GCS, Azure Blob.

## 주요 특성

- **객체 단위 저장**: 파일을 키(이름)로 PUT/GET. 부분 수정이 아니라 통째 쓰기/읽기.
- **사실상 무한 확장**: 용량을 신경 쓰지 않고 페타바이트급까지.
- **저비용·고내구성**: 다중 복제로 11 9's 수준 내구성, 저장 단가가 블록/DB보다 저렴.
- **HTTP 접근 + [[pre-signed-url]]**: 클라이언트 직접 업로드/다운로드.
- **flat 네임스페이스**: 진짜 디렉터리가 아니라 키 prefix로 계층 흉내.

## 언제 선택하는가 / 대안 비교

| 후보 | 단위 | 적합 | 부적합 |
|---|---|---|---|
| **Blob/Object** | 객체(파일) | 큰 비정형 파일·미디어·백업 | 부분 갱신·저지연 랜덤 IO |
| Block storage(EBS) | 블록 | DB·OS 디스크·낮은 지연 | 글로벌 공유·대용량 정적 |
| File storage(NFS) | 파일+디렉터리 | 공유 파일시스템 | 초대규모 확장 |
| [[relational-database]] | 행 | 구조화·트랜잭션 | 대용량 바이너리(안티패턴) |

핵심 판단: **큰 파일을 통째로 읽고 쓰며 메타데이터는 DB에 따로** 두는 구조면 blob storage. 영상 본체는 blob, URL·크기·해상도 같은 메타는 [[relational-database]]/캐시에 — ch14의 분리 설계가 전형.

## 전형적 사용 사례

- 영상·이미지·오디오 원본 및 인코딩 산출물 (ch14).
- 정적 자산의 origin([[cdn]]이 앞단 캐시).
- 로그·데이터 레이크·백업·아카이브.

## 실무 함정

- **부분 수정 불가**: 객체 일부만 바꾸려면 통째 재업로드 — 잦은 소규모 갱신엔 부적합.
- **일관성 모델**: 과거엔 eventual이었으나 현재 S3 등은 read-after-write 강일관성 제공 — 버전·세대별 보장 확인 필요.
- **요청·전송 비용**: 저장은 싸도 GET/전송·CDN out 비용이 누적([[cdn]] 비용이 ch14의 핵심 문제).
- 메타데이터를 blob에 의존하지 말 것 — 질의가 필요한 메타는 DB로 분리.
