# Document Database

## 한 줄 정의

데이터를 **자기 기술적(self-describing) 문서**(JSON/BSON 등) 단위로 저장하는 NoSQL 데이터베이스. 스키마가 유연하고 중첩 구조를 그대로 담을 수 있어, 직렬화된 객체·반정형 데이터에 적합하다. 대표: MongoDB (ch13, p.212).

## 주요 특성

- **유연한 스키마**: 문서마다 필드가 달라도 됨. 사전 스키마 정의 불필요.
- **중첩 문서·배열**: 객체 그래프를 join 없이 한 문서에 임베드.
- **문서 단위 원자성**: 한 문서 내 갱신은 원자적(문서 간 트랜잭션은 제한적이거나 별도 지원).
- **2차 인덱스·쿼리**: KV store보다 풍부한 질의(필드 조건·정렬).

## 언제 선택하는가 / 대안 비교

| 후보 | 모델 | 강점 | 약점 |
|---|---|---|---|
| **Document DB** (MongoDB) | JSON 문서 | 유연 스키마·중첩·풍부한 질의 | 다문서 트랜잭션·복잡 join 약함 |
| [[relational-database]] | 행/테이블 | 강한 일관성·join·트랜잭션 | 스키마 경직·수평 확장 어려움 |
| [[nosql-database]] KV | key→value | 최저 지연·단순 확장 | 질의력 빈약(키 조회만) |

핵심 판단: **데이터가 문서 형태(반정형·중첩)이고 스키마가 자주 변하면** document DB. 강한 트랜잭션·복잡 join이 본질이면 RDB. 단순 key 조회로 충분하면 KV가 더 가볍다.

## 전형적 사용 사례

- 직렬화된 객체 스냅샷 저장 (ch13: 주간 빌드된 trie를 직렬화해 보관).
- 콘텐츠 관리·카탈로그·사용자 프로필처럼 필드가 가변적인 데이터.
- 빠른 프로토타이핑(스키마 마이그레이션 부담↓).

## 실무 함정

- **유연 스키마의 양날**: 검증을 앱이 떠안아 데이터 품질이 흐트러지기 쉽다. 스키마 검증을 명시적으로 둘 것.
- 비정규화·임베드가 읽기엔 좋지만 **중복 갱신**이 늘어난다(같은 데이터를 여러 문서가 복사).
- 다문서 트랜잭션은 비싸거나 제약 — 경계를 넘는 일관성이 필요하면 모델을 재검토.
- 큰 문서·무한 성장 배열은 안티패턴(문서 크기 한계·이동 비용).
